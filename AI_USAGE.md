# AI Usage

## Инструменты

- Claude (чат): архитектура, спецификации для backend и frontend, код-ревью, этот документ.
- Claude Code: реализация в двух отдельных сессиях, backend и frontend, каждая по своей спецификации.

## Что делегировано

Архитектуру, контракты API, модель данных и классификацию ошибок я проектировал сам, до начала работы агента, и оформлял в спецификациях: документах с описанием архитектуры и пошаговым планом по фазам, где для каждой фазы задан конкретный Definition of Done. Агент читал спецификацию целиком и дальше реализовывал её по фазам, не переходя к следующей без моего подтверждения, что предыдущая прошла проверку.

## Примеры промптов

1. Старт backend-сессии: прочитать спецификацию, предложить план по Фазе 0, дождаться подтверждения, при поводе отклониться от плана останавливаться и спрашивать, а не решать самому.
2. Старт frontend-сессии: сначала посмотреть реальную структуру репозитория (куда класть код рядом с backend), контракт из спецификации считать финальным источником истины.
3. Добавление RealProvider: точечное аддитивное изменение (новый файл, одна зависимость, одна ветка в DI), без права трогать существующий FakeProvider, decode_service и тесты.

## Принято / изменено / отклонено

- Принято как есть: FakeProvider с детерминированными маркерами ошибок в тексте брифа, без обращения к реальному API.
- Изменено: первая версия decode_service дублировала одну и ту же пару строк (mark_failed + return) четыре раза и держала код и текст ошибки в двух не связанных друг с другом местах. Свёл в одну функцию, возвращающую код и сообщение вместе, и локальную функцию fail() вместо повтора.
- Отклонено: свой же черновик спецификации, где комментарии в коде разрешались на русском. Поправил до старта реализации: английский как отраслевой стандарт независимо от языка команды.

## Как проверял

На каждой фазе, помимо прогона проверок, делал ревью сгенерированного кода построчно, не только смотрел на зелёные тесты. После каждой фазы гонял руками: curl по всем сценариям ошибок, содержимое таблицы через psql, полный pytest, чистая пересборка docker compose up --build, реальная загрузка prod-сборки расширения в chrome://extensions с проверкой всех состояний UI. RealProvider проверен одним запросом с реальным ключом Gemini (бесплатный тариф), ключ отозван сразу после проверки.

## Найденная проблема в выводе агента

Дублирование кода в decode_service (см. выше). Тесты проходили у обеих версий, но код требовал обычного ревью, не только зелёных тестов.
